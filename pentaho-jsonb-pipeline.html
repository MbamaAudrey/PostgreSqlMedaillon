1. Structure de la table Bronze et les 5 endpoints

Votre table bronze a deux colonnes. La colonne endpoint identifie le type de donnée, la colonne data contient le JSON brut :

CREATE TABLE bronze.raw_data ( id SERIAL PRIMARY KEY, endpoint VARCHAR(200), -- identifie le type de données data JSONB, -- contenu brut loaded_at TIMESTAMP DEFAULT NOW() );

Les cinq valeurs possibles du champ endpoint :

endpoint (valeur exacte)ContenuClé principale
v5.8.0/party/customersClientsheader.id
v9.4.0/holdings/accounts/currentAccountsComptes courantsheader.id = numéro de compte
v1.0.1/order/transfersVirementsheader.id = ID transaction
v9.0.0/loansPrêts$.id
v10.0.0/depositsDépôts$.id
Principe de filtrage : chaque Transformation commence par un Entrée de table qui filtre sur endpoint = 'valeur_exacte'. Vous obtenez ainsi un flux de lignes contenant uniquement les enregistrements du bon type. Ensuite vous extrayez les champs JSONB avec le step Valeur JSON.

2. Principe : extraction JSONB dans Pentaho

Pour extraire des champs d'une colonne JSONB, vous avez deux approches. Voici laquelle choisir selon le cas :

ApprocheCommentQuand l'utiliser
Option A — SQL PostgreSQL Extraire directement dans le SQL du step "Entrée de table" avec les opérateurs ->> et -> de PostgreSQL Recommandé — plus simple, plus rapide, PostgreSQL fait le travail
Option B — Step "Valeur JSON" Pentaho lit la colonne JSONB comme texte, puis le step "Valeur JSON" (catégorie : Transformation) extrait les champs via JSONPath Utile si le SQL devient trop complexe (tableaux imbriqués, structures variables)
On utilise l'Option A (SQL) dans ce guide. C'est la plus simple : vous écrivez le SQL d'extraction dans le step "Entrée de table", et Pentaho reçoit directement des colonnes typées — pas de JSON à parser dans Pentaho.

Rappel des opérateurs PostgreSQL JSONB :

OpérateurRésultatExemple
->Retourne un objet JSONBdata -> 'body'
->>Retourne du texte (pour extraire une valeur finale)data ->> 'status'
-> 'key' ->> 'key2'Navigue dans les niveaux imbriquésdata -> 'header' ->> 'id'
-> 'arr' -> 0 ->> 'key'Accède à l'index 0 d'un tableaudata -> 'body' -> 'displayNames' -> 0 ->> 'displayName'

6. Prêts → silver.prets

Fichier : transfo_bronze_prets_to_silver.ktr

Structure différente — champs à la racine du JSON (pas de header/body)
SELECT data ->> 'id' AS id_pret, data ->> 'status' AS statut, data ->> 'transactionStatus' AS statut_transaction, loaded_at FROM bronze.raw_data WHERE endpoint = 'v9.0.0/loans' AND data ->> 'status' = 'success' AND data ->> 'transactionStatus' = 'Live'
Pour les prêts, les champs sont directement à la racine du JSON ($.id, $.status) — pas de niveau header ou body. On utilise donc data ->>'id' directement, sans navigation imbriquée.

Suite identique aux autres : Filtre de lignes sur id_pret IS NOT NULL, puis Insertion / Mise à jour dans silver.prets avec id_pret comme clé.

7. Dépôts → silver.depots

Fichier : transfo_bronze_depots_to_silver.ktr

Structure identique aux prêts — champs à la racine
SELECT data ->> 'id' AS id_depot, data ->> 'status' AS statut, loaded_at FROM bronze.raw_data WHERE endpoint = 'v10.0.0/deposits' AND data ->> 'status' = 'success'
Données minimales : les dépôts n'ont que id et status dans la documentation actuelle. Si votre JSONB réel contient plus de champs, ajoutez-les dans le SELECT avec la même syntaxe data ->>'champ'.
